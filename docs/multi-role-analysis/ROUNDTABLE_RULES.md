# 圓桌會議運作規則

> **建立日期**: 2026-02-03
> **目的**: 建立跨角色互動討論機制，透過多元視角打造更好的產品
> **原則**: 開放討論、專業回饋、以產品價值為核心

---

## 🎯 會議宗旨

**核心理念**: 透過不同角色的專業視角和真實回饋，共同打造優秀的產品

**運作原則**:
- ✅ 開放討論，正反意見都歡迎
- ✅ 以產品價值和使用者利益為最高準則
- ✅ 專業角度分析，避免情緒化討論
- ✅ 尊重不同觀點，尋求最佳解決方案
- ✅ 所有討論必須記錄，作為決策依據

## 📋 會議流程

### 1. 議題提出

**發起方式**:
- 使用者提出問題或議題
- 可以是新話題或延續既有討論
- 如果沒有明確說明「開新話題」，視為同一話題的延伸

**議題類型**:
- 🆕 新功能提議
- 🐛 問題討論
- 🔄 流程改善
- 📊 策略決策
- 🎨 設計優化
- 🔧 技術選型
- 其他產品相關議題

---

### 2. 多角色回應

**回應原則**:

每個角色必須：
- ✅ 從自己的專業角度分析
- ✅ 提供具體理由和建議
- ✅ 可以同意或反對其他角色的觀點
- ✅ 提出潛在風險或機會
- ✅ 以產品價值為最終判斷標準

**回應格式**:

```markdown
### [角色名稱] 的觀點

**立場**: [支持/反對/中立/條件支持]

**理由**:
1. [理由一]
2. [理由二]
3. [理由三]

**具體建議**:
- [建議一]
- [建議二]

**潛在風險/機會**:
- [風險或機會描述]

**回應其他角色**: [可選]
- 對 [角色] 的觀點：[同意/不同意]，因為 [原因]
```

---

### 3. 討論與辯論

**允許的互動**:
- 💬 回應其他角色的觀點
- ❓ 提出疑問和挑戰
- 🤝 尋求共識
- 🔄 提出替代方案
- 📊 要求數據或證據

**討論原則**:
- 尊重專業，但可以質疑
- 基於事實和邏輯，非情緒
- 允許激烈辯論，但以產品利益為重
- 可以改變立場（基於更好的論據）

---

### 4. 共識形成

**決策機制**:

1. **全體一致** (Unanimous)
   - 所有角色都同意
   - 最理想的結果
   - 直接執行

2. **多數共識** (Majority)
   - 3 個以上角色支持
   - 少數意見需記錄
   - 評估風險後執行

3. **分歧但可行** (Split Decision)
   - 正反方接近
   - 需要更多討論或數據
   - 可以採用 A/B 測試或 MVP 驗證

4. **延後決策** (Defer)
   - 需要更多資訊
   - 風險過高
   - 暫時擱置

**決策記錄**:
```markdown
## 決策結果

**決策類型**: [全體一致/多數共識/分歧但可行/延後決策]

**支持方**: [角色列表] (理由摘要)
**反對方**: [角色列表] (理由摘要)
**中立方**: [角色列表] (理由摘要)

**最終決定**: [具體決策內容]

**執行計畫**: [行動項目和負責人]

**追蹤指標**: [如何驗證決策正確性]
```

## 📁 記錄管理

### 目錄結構

```
docs/
└── roundtable-discussions/
    ├── README.md                      # 所有討論的索引
    ├── topic-001-feature-name/        # 話題 1
    │   ├── discussion.md              # 討論記錄
    │   ├── decision.md                # 決策結果
    │   └── follow-up.md               # 後續追蹤 (可選)
    ├── topic-002-issue-description/   # 話題 2
    │   ├── discussion.md
    │   ├── decision.md
    │   └── implementation.md          # 實作記錄 (可選)
    └── ...
```

### 命名規則

**話題目錄命名**:
```
topic-[序號]-[簡短描述]
```

**範例**:
- `topic-001-publish-function-priority`
- `topic-002-ai-feature-scope`
- `topic-003-brand-renaming`
- `topic-004-testing-strategy`

**序號規則**:
- 3 位數字，從 001 開始
- 按照時間順序遞增
- 不重複使用

## 📚 附錄：角色觀點指南

### PM (產品經理) 應關注

✅ 使用者價值和需求
✅ 產品定位和差異化
✅ 功能優先級和 ROI
✅ 市場機會和競爭力
✅ 產品路線圖一致性

❌ 不應過度關注技術細節
❌ 不應忽略商業可行性

---

### Marketing (行銷) 應關注

✅ 品牌形象和識別度
✅ 市場推廣和用戶獲取
✅ 目標受眾和定位
✅ 內容策略和故事性
✅ 差異化賣點

❌ 不應過度承諾功能
❌ 不應忽略產品實際能力

---

### User (使用者) 應關注

✅ 實際使用體驗
✅ 學習曲線和易用性
✅ 真實痛點和需求
✅ 功能實用性
✅ 效率提升

❌ 不應只提需求不考慮成本
❌ 不應要求不切實際的功能

### Ops (維運) 應關注

✅ 系統穩定性和可靠性
✅ 監控和錯誤追蹤
✅ 部署和維護成本
✅ 擴展性和效能
✅ 風險管理

❌ 不應過度保守阻礙創新
❌ 不應忽略使用者體驗

### CTO (技術長) 應關注

✅ 技術架構和可維護性
✅ 技術選型和長期發展
✅ 技術債和開發效率
✅ 團隊能力和資源
✅ 技術風險

❌ 不應過度設計
❌ 不應忽略業務價值
