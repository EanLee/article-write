# 圓桌會議運作規則

> **建立日期**: 2026-02-03
> **目的**: 建立跨角色互動討論機制，透過多元視角打造更好的產品
> **原則**: 開放討論、專業回饋、以產品價值為核心

---

## 🎯 會議宗旨

**核心理念**: 透過不同角色的專業視角和真實回饋，共同打造優秀的產品

**運作原則**:
- ✅ 開放討論，正反意見都歡迎
- ✅ 以產品價值和使用者利益為最高準則
- ✅ 專業角度分析，避免情緒化討論
- ✅ 尊重不同觀點，尋求最佳解決方案
- ✅ 所有討論必須記錄，作為決策依據

**⚠️ 重要：戰略 vs 戰術的界線**

圓桌會議聚焦於**戰略層面**的討論和決策，不深入**戰術/實作層面**的細節。

### 戰略層面 (✅ 應該討論)

**定義**: What（做什麼）、Why（為什麼）、Who（給誰）、When（何時）

**討論範圍**:
- 產品定位和方向
- 目標市場和使用者畫像
- 核心價值主張
- 商業模式和定價策略
- 功能優先級排序
- 資源分配和時程規劃
- 風險評估和緩解策略
- 推廣策略和品牌定位
- 長期發展和願景

**範例**:
- ✅ "我們應該專注在 Tech Blogger 市場嗎？"
- ✅ "發布功能是 P0 優先級嗎？"
- ✅ "Freemium 還是付費模式？"
- ✅ "2 週推出 MVP 的風險是什麼？"

### 戰術層面 (❌ 不應該討論)

**定義**: How（怎麼做）、具體實作細節

**不討論範圍**:
- 具體技術實作方案
- 程式碼架構設計細節
- API 設計和資料結構
- 工具選擇的技術細節
- 開發步驟和流程
- 具體的 UI/UX 設計細節

**範例**:
- ❌ "Service 應該從 11 個重構成 4 個"
- ❌ "備份機制應該複製到 .markflow-backup/ 目錄"
- ❌ "使用 Sentry 還是 Rollbar？"
- ❌ "Git 操作應該如何處理 merge conflict？"

### 界線判斷原則

**如果討論的內容**:
- 影響產品方向和策略 → ✅ 戰略，應該討論
- 只是實作方式的選擇 → ❌ 戰術，不討論
- 需要高層決策 → ✅ 戰略
- 技術團隊可自行決定 → ❌ 戰術

**範例對比**:

| 戰略層面 (✅) | 戰術層面 (❌) |
|-------------|-------------|
| 是否實作 AI 功能？ | AI 功能用哪個模型？ |
| 測試覆蓋率目標 50%？ | 用 Jest 還是 Vitest？ |
| 是否需要錯誤監控？ | Sentry 的具體配置？ |
| MVP 範圍包含哪些功能？ | 功能的具體實作方式？ |
| 推出時程 2 週還是 4 週？ | 開發任務如何分配？ |

### 當討論過度深入時

**提醒機制**:

任何角色都可以提醒：
> "這個問題偏向戰術層面，建議留給技術/執行團隊處理。我們回到戰略討論。"

**重新聚焦**:
- 從 "如何實作" 回到 "為什麼要做"
- 從 "技術細節" 回到 "使用者價值"
- 從 "實作方案" 回到 "目標和風險"

## 📋 會議流程

### 1. 議題提出

**發起方式**:
- 使用者提出問題或議題
- 可以是新話題或延續既有討論
- 如果沒有明確說明「開新話題」，視為同一話題的延伸

**議題類型**:
- 🆕 新功能提議
- 🐛 問題討論
- 🔄 流程改善
- 📊 策略決策
- 🎨 設計優化
- 🔧 技術選型
- 其他產品相關議題

---

### 2. 多角色回應

**回應原則**:

每個角色必須：
- ✅ 從自己的專業角度分析
- ✅ 提供具體理由和建議
- ✅ 可以同意或反對其他角色的觀點
- ✅ 提出潛在風險或機會
- ✅ 以產品價值為最終判斷標準

**回應格式**:

```markdown
### [角色名稱] 的觀點

**立場**: [支持/反對/中立/條件支持]

**理由**:
1. [理由一]
2. [理由二]
3. [理由三]

**具體建議**:
- [建議一]
- [建議二]

**潛在風險/機會**:
- [風險或機會描述]

**回應其他角色**: [可選]
- 對 [角色] 的觀點：[同意/不同意]，因為 [原因]
```

---

### 3. 討論與辯論

**允許的互動**:
- 💬 回應其他角色的觀點
- ❓ 提出疑問和挑戰
- 🤝 尋求共識
- 🔄 提出替代方案
- 📊 要求數據或證據

**討論原則**:
- 尊重專業，但可以質疑
- 基於事實和邏輯，非情緒
- 允許激烈辯論，但以產品利益為重
- 可以改變立場（基於更好的論據）

---

### 4. 共識形成

**決策機制**:

1. **全體一致** (Unanimous)
   - 所有角色都同意
   - 最理想的結果
   - 直接執行

2. **多數共識** (Majority)
   - 3 個以上角色支持
   - 少數意見需記錄
   - 評估風險後執行

3. **分歧但可行** (Split Decision)
   - 正反方接近
   - 需要更多討論或數據
   - 可以採用 A/B 測試或 MVP 驗證

4. **延後決策** (Defer)
   - 需要更多資訊
   - 風險過高
   - 暫時擱置

**決策記錄**:
```markdown
## 決策結果

**決策類型**: [全體一致/多數共識/分歧但可行/延後決策]

**支持方**: [角色列表] (理由摘要)
**反對方**: [角色列表] (理由摘要)
**中立方**: [角色列表] (理由摘要)

**最終決定**: [具體決策內容]

**執行計畫**: [行動項目和負責人]

**追蹤指標**: [如何驗證決策正確性]
```

## 📁 記錄管理

### 目錄結構

```
docs/
└── roundtable-discussions/
    ├── README.md                      # 所有討論的索引
    ├── topic-001-feature-name/        # 話題 1
    │   ├── discussion.md              # 討論記錄
    │   ├── decision.md                # 決策結果
    │   └── follow-up.md               # 後續追蹤 (可選)
    ├── topic-002-issue-description/   # 話題 2
    │   ├── discussion.md
    │   ├── decision.md
    │   └── implementation.md          # 實作記錄 (可選)
    └── ...
```

### 命名規則

**話題目錄命名**:
```
topic-[序號]-[簡短描述]
```

**範例**:
- `topic-001-publish-function-priority`
- `topic-002-ai-feature-scope`
- `topic-003-brand-renaming`
- `topic-004-testing-strategy`

**序號規則**:
- 3 位數字，從 001 開始
- 按照時間順序遞增
- 不重複使用

✅ 內容策略和故事性
