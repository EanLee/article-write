# 圓桌會議運作規則

> **建立日期**: 2026-02-03
> **最後更新**: 2026-02-14
> **版本**: v2.0
> **目的**: 建立跨角色討論與決策機制，透過多元視角打造更好的產品

---

## 🎯 會議定位

### 圓桌會議 vs 技術會議

| 面向 | 圓桌會議 | 技術會議 |
|------|---------|---------|
| 層次 | 戰略（What / Why） | 戰術（How / 實作細節） |
| 參與者 | 全體角色（五個角色） | 技術團隊成員 |
| 討論內容 | 產品方向、功能優先級、使用者價值、風險評估 | 實作方式、架構設計、程式細節 |
| 程式碼 | ❌ 不出現（除非特別需要上層決策的技術議題） | ✅ 正常出現 |
| 文件位置 | `docs/roundtable-discussions/` | `docs/tech-team/` |

### 戰略層面（✅ 圓桌討論）

- 產品定位與方向
- 功能優先級與 MVP 範圍
- 使用者價值與痛點
- 商業模式與推廣策略
- 跨團隊影響的重大決策
- 風險評估（影響產品方向層級）

### 戰術層面（❌ 技術會議處理）

- 具體技術實作方案
- 程式碼架構與資料結構
- API 設計、工具選型細節
- 開發步驟與任務分配

### 升級機制：技術會議 → 圓桌

**任何人**在技術實作過程中，若發現：
- 風險影響到產品方向或使用者體驗
- 需要多人共同決策而非技術團隊自行決定
- 實作結果與原本的戰略假設有落差

→ **主動發起圓桌會議**，無需等待上級指示。

**發起前提**：**必須先準備討論材料**（議題說明、背景、選項、影響評估），不得空手發起。

## 📋 發起流程

### 誰可以發起

**任何人**皆可發起圓桌會議，包含：
- 使用者（產品需求、功能想法）
- 技術團隊成員（發現實作風險影響策略）
- 任何角色成員

### 發起前必須準備

發起前**必須先備妥討論材料**，包含：

1. **議題說明**：要討論什麼、為什麼需要圓桌決策
2. **背景資訊**：相關歷史脈絡、現況說明
3. **討論選項**（若有）：可能的方向或方案
4. **影響評估**：對產品、使用者、時程的潛在影響

未準備材料者，請先整理後再發起。

## 📁 文件結構與命名

### 目錄結構

```
docs/roundtable-discussions/
├── README.md                                    ← 所有議題索引
├── .template/
│   ├── discussion.md                            ← 討論記錄模板
│   └── decision.md                              ← 決策記錄模板
├── topic-001-2026-02-03-product-launch/         ← 議題資料夾
│   ├── discussion.md                            ← 討論過程（完整記錄）
│   └── decision.md                              ← 決策結果（結論摘要）
├── topic-002-2026-02-06-progress-review/
│   ├── discussion.md
│   └── decision.md
└── ...
```

### 資料夾命名規則

```
topic-NNN-YYYY-MM-DD-簡短描述/
```

| 部分 | 說明 |
|------|------|
| `topic` | 固定前綴 |
| `NNN` | 三位數序號，從 001 起，全域唯一，反映時間順序 |
| `YYYY-MM-DD` | 發起日期 |
| `簡短描述` | 英文小寫，以 `-` 分隔，簡短描述議題 |

**範例**：
```
topic-006-2026-02-14-publish-mechanism/
topic-007-2026-02-14-frontmatter-date-fields/
```

同一天多個議題透過序號（NNN）區分前後順序。

### 兩份文件的分工

| 文件 | 內容 | 用途 |
|------|------|------|
| `discussion.md` | 完整討論過程，包含各角色論點、交叉回應、分歧點 | 了解決策脈絡 |
| `decision.md` | 決策結果摘要，包含行動項目、負責人 | 快速查看結論 |

## 🔄 決策機制

| 類型 | 條件 | 處理方式 |
|------|------|---------|
| 全體一致 | 五角色全部同意 | 直接執行 |
| 多數共識 | 三角色以上支持 | 記錄少數意見後執行 |
| 分歧但可行 | 正反接近 | 追加討論或小規模驗證 |
| 延後決策 | 資訊不足或風險過高 | 列出待釐清項目，排期再議 |

---

## ⚠️ 禁止事項

- ❌ 討論記錄出現程式碼（除非議題本身需要上層決策的技術選項）
- ❌ 任何角色缺席（五個角色必須全部出現）
- ❌ 空手發起（未準備討論材料）
- ❌ 在圓桌討論戰術細節（應移至技術會議）
- ❌ 決策後不更新 decision.md
