## 💬 討論記錄

### 第一輪：各角色直言

---

#### 💻 CTO Taylor 的觀點

**立場**: ⚠️ 嚴重警示

**不好的地方**:
1. **319 個單元測試，0 個端對端整合測試**。每個 Service 單獨正確，但整體流程是否跑通沒有驗證過。這是技術面最大的盲點。
2. **技術文件沒有被整合進工作流程**。T-002 的技術評估報告寫得完整，但差點沒被參考——False Positive 修復是被明確提醒後才補上的。文件存在不等於文件被用到。

**改善建議**:
- 端對端驗證要成為每個 Sprint 的 Gate，不是可選項
- 技術評估文件在開始實作前要被 check 一遍，納入 PR checklist

---

#### 👤 User Jordan 的觀點

**立場**: ❌ 強烈不滿

**不好的地方**:
- **產品沒有可安裝的版本**。319 個測試通過，但我一直只能看 localhost。對我來說，功能再多等於零，因為我根本沒辦法「用」。
- 開發進度服務「技術正確性」，沒有服務「用戶可以跑起來」這件事。

**改善建議**:
- **每個 Sprint 的 Definition of Done 必須包含「可安裝的版本」**。哪怕很陽春，但要有 artifact 可以真正執行。
- 沒有可用的 artifact，Sprint 等於沒完成。

---

#### 🎯 PM Alex 的觀點

**立場**: ⚠️ 有具體問題要解決

**不好的地方**:
- **沒有統一的進度可視化**。T-001 任務 8 掛了好幾輪「🔄 下一步」，但 E2E 文件早就更新完了。決策記錄和實作狀態散落各處，連自己人都不清楚 MVP 做了幾成。
- 「Done」的定義沒有統一，導致每個人的完成標準不一樣。

**改善建議**:
- 建立 `MVP_STATUS.md`，一頁清單列出 MVP 所有功能、Done 定義、當前狀態
- 不要讓進度散落在各個 decision.md 和 T-NNN 文件裡

---

#### 📢 Marketing Lisa 的觀點

**立場**: ⚠️ 擔憂長期推廣能力

**不好的地方**:
- **整個開發過程沒有生成任何可用的行銷素材**。沒有截圖、沒有 demo 影片、沒有可安裝版本。如果今天有人問「WriteFlow 長什麼樣子」，我只能給文字描述。
- 「做著做著完全忘記最終要給人看」這個心態是問題。技術上很嚴謹，但對外完全不可見。

**改善建議**:
- **每個 Sprint 至少一張截圖**。功能完成時截圖，放進 `docs/screenshots/`。成本極低，長期累積有大價值。
- 不是要求現在推廣，但要開始有素材的意識。

---

#### 🔧 Ops Sam 的觀點

**立場**: ❌ 嚴重設計缺失

**不好的地方**:
1. **錯誤路徑完全未設計**。如果 `syncAllPublished()` 在用戶端失敗——網路中斷、檔案權限不足、目標路徑不存在——用戶看到什麼？目前是 `console.error`，什麼都沒有。
2. **維運視角完全缺席**。翻遍 T-001、T-002、所有 decision.md，沒有一份文件討論「如何處理失敗」。我們的開發文化是「讓 happy path 跑通」，sad path 不在 Done 的定義裡。

**改善建議**:
1. PublishService 的錯誤要有**明確分類和有意義的用戶反饋**——不是 `console.error`，是「第 3 篇文章失敗，原因：目標路徑不存在」。
2. 每個新功能的 PR 要有**「失敗路徑清單」**，列出可能失敗的情況和處理方式。

### 第二輪：跨角色回應

**Jordan 回應 Sam**：
Sam 說的錯誤反饋問題我很有感。之前測試同步功能，沒反應，完全不知道是成功還是失敗。這種不確定感很讓人焦慮，甚至讓人不敢用。

**Taylor 回應 Lisa**：
截圖的建議成本幾乎為零，完全支持。但要澄清：打包沒有完成是因為 electron-builder 設定還沒走過完整 build pipeline，不是忘記——是排序問題。打包被排在「功能完成後」，但功能的邊界從來沒有被清楚定義，所以打包一直被推後。

**Lisa 回應 Taylor**：
我接受這個解釋，但「打包等功能完成後」這個排序本身就是問題。對我們來說打包是「最後一步」，但對使用者來說打包是「第一步」。這個優先序的錯位才是真正需要修正的。

**Alex 回應所有人**：
我聽到了四個不同層次的瓶頸：（1）可交付物缺失，（2）進度可視化缺失，（3）失敗路徑缺失，（4）文件與工作流程脫節。這四個問題有一個共通的根因——**我們的 Done 定義太鬆了**。只要程式碼跑起來、測試通過就算完成，但沒有涵蓋「打包」、「錯誤處理」、「用戶可操作」這些維度。

**Sam 回應 Alex**：
完全同意。而且 Git 靜默自動化這件事是最具體的證明——圓桌 #005 決定了要做，但因為沒有人的 Sprint 把它列為 P0，它就一直飄著。這是優先級管理的問題，不是能力問題。

## 🤝 共識與分歧

### 共識點
- 下一個 Sprint 的最高優先是「讓 Jordan 真正可以跑起來」，不加新功能
- 打包（electron-builder）必須做
- 錯誤反饋 UI 必須補
- `MVP_STATUS.md` 建立，統一進度視圖
- 截圖規範納入流程

### 主要分歧
- **Git 自動化是否現在做**：Sam 和 Jordan 認為 v0.2 的計畫可維持，Alex 確認圓桌 #005 的決策不變——**MVP 不包含 Git 自動化**，v0.2 再做。

### 核心瓶頸識別

| 瓶頸 | 根因 | 類型 |
|------|------|------|
| 無可交付物（打包）| 打包未被列為 Sprint P0 | 優先序問題 |
| 進度不透明 | Done 定義鬆散，狀態散落各處 | 流程問題 |
| 錯誤路徑缺失 | 開發文化 happy path 導向 | 文化問題 |
| 文件與實作脫節 | 文件存在但未整合進工作流程 | 工具問題 |
