## 💬 Part 2：各角色直言

---

### Jordan（User）

**Jordan**：我先說，因為我的問題最直接。

這個產品**現在沒有辦法安裝**。沒有 `.exe`，沒有 `.app`，我只能在你們的開發環境裡看 localhost。對我來說，你們做了什麼、測試通過了多少個，這些我感受不到。我感受到的只有一件事：我沒辦法用。

不好在哪裡？就是這件事從來沒有被當成優先項目。整個開發過程我一直被告知「功能在做了」，但「能不能讓我跑起來」這件事好像一直不急。

我的改善建議是：**每個 Sprint 的完成定義，必須包含一個可以安裝的版本**。哪怕功能少，哪怕很陽春，但要有東西我可以真正點開來用。沒有 artifact，那個 Sprint 對我來說等於沒完成。

---

### Alex（PM）

**Alex**：Jordan 說的是結果，但我想說根因是什麼。

我觀察到一個問題——**我們沒有統一的「完成」定義**。T-001 的任務 8，E2E 文件更新，掛著「🔄 下一步」掛了好幾輪，但那份文件其實早就更新好了。沒有人去把狀態打勾，因為沒有人覺得這件事「必須」做。

更大的問題是：現在的任務進度散落在 T-001、T-002、各個 `decision.md` 裡面。要知道 MVP 做了幾成，要翻好幾個文件，自己拼起來。這不合理。

我的改善建議：**建立一個 `MVP_STATUS.md`，一頁清單**，列出 MVP 的所有功能，每個功能有明確的 Done 定義，有當前狀態。不要讓人自己去拼進度。

---

### Sam（Ops）

**Sam**：我要說一個大家可能沒注意到的問題。

你們知道如果 `syncAllPublished()` 在用戶端失敗了，用戶會看到什麼嗎？`console.error`。就這樣。目標路徑不存在、磁碟空間不足、檔案權限不夠——對用戶來說全都是黑盒子。

翻遍 T-001、T-002、所有的 decision 文件，我找不到任何一個地方討論過「如果失敗了怎麼辦」。我們的整個開發文化是：**讓 happy path 跑通，sad path 不在考慮範圍裡**。

這在開發環境裡看不出問題，因為開發環境永遠是理想狀態。但用戶的環境不是——路徑可能是錯的，權限可能不夠，磁碟可能滿了。

我的改善建議分兩個。第一：`PublishService` 的錯誤處理要有**有意義的用戶反饋**，不是 `console.error`，是「第 3 篇文章失敗，原因：目標路徑不存在，請到設定確認部落格路徑」這種程度。第二：**每個 PR 要有「失敗路徑清單」**，逼自己想一遍這個功能可能在哪裡壞掉，怎麼處理。

---

**Jordan**（打斷）：等等，Sam 你說的錯誤反饋，我超有感！我之前測試同步，按下去沒反應，我完全不知道是成功了還是失敗了。我等了三十秒以為是在跑，後來發現根本什麼都沒動。那種感覺很差，甚至讓我不敢再按。

**Sam**：對，這就是我說的問題。沒有反饋等於用戶不信任這個功能。

### Lisa（Marketing）

**Lisa**：我說一個大家可能不想聽的。

**我們現在沒有任何可以拿出去給人看的東西**。沒有截圖、沒有 demo 影片、沒有安裝包。如果今天有人問我「WriteFlow 長什麼樣子」，我能給的只有文字描述。在 2026 年，這是不夠的。

不好的地方是：整個開發過程，大家都在想「功能做對了嗎」，但沒有人在想「做完的東西有沒有被看見」。不是說要現在推廣，但「做著做著完全忘記最終是要給人看的」——這個心態本身是問題。

我的改善建議很簡單：**每個功能完成之後截一張圖，放進 `docs/screenshots/`**。成本是五秒鐘，但長期累積下來，對我是行銷素材，對開發是進度可視化，對所有人都有好處。

---

### Taylor（CTO）

**Taylor**：我來說兩個技術面的問題。

第一：**319 個單元測試，但 0 個端對端整合測試**。每個 Service 單獨是對的，但整個發布流程從頭到尾有沒有跑通，沒有人驗證過。這是最大的技術盲點。單元測試在完美環境裡跑，不代表真實的串接沒問題。

第二：**技術文件存在，但沒有進入工作流程**。T-002 的技術評估報告分析得很詳細，但它差點沒被參考到——False Positive 的修復是在被明確提醒之後才補上的。如果文件要有意義，它必須在對的時間點被看到，不能只是存在那裡等人主動去翻。

## 📊 Part 4：收斂

**Taylor**：好，我整理一下今天的結論。

我們識別出四個瓶頸：

第一，**沒有可交付物**——打包從來不是任何一個 Sprint 的 P0。根因是「打包是最後一步」這個錯誤的排序假設。

第二，**進度不透明**——Done 定義鬆散，狀態散落各處。根因是沒有統一的進度視圖。

第三，**錯誤路徑缺失**——happy path 跑通就算完成，sad path 不在 Done 裡面。根因是開發文化問題，沒有機制逼人想失敗情境。

第四，**文件與工作流程脫節**——技術評估文件存在，但沒有在正確時機被引用。根因是文件沒有入 checklist。

**下一個 Sprint 的方向**，全體確認一下：不加任何新功能，只做「讓 Jordan 真正跑起來」——打包、補錯誤反饋、端對端驗證。

**Jordan**：這是我最需要的。

**Alex**：支持。比加新功能更重要。

**Lisa**：同意。打包之後我需要截圖。

**Sam**：打包之前要先把錯誤反饋補好，不然 Jordan 跑起來遇到問題沒有反饋，第一印象就壞了。

**Taylor**：Sam 說得對，順序是：補錯誤反饋 → 打包 → Smoke Test → Jordan 驗收。

行動項目我整理好之後放到 decision.md。今天到這裡。
