## 討論記錄

### 第一輪：開場——各角色最在意的文章要素（2026-02-27）

**Casey（SEO 專家）**：「讓我直接講重點。一篇文章發出去之後，Google 怎麼看它，決定了 80% 的自然流量。從 SEO 角度，我第一眼看的是：標題有沒有關鍵字、URL Slug 夠不夠簡潔精準、Meta Description 有沒有在 120-160 字之間、H1 只有一個嗎？這些是基礎。更深一層，我會看文章結構——H2/H3 層次有沒有邏輯、有沒有 Internal Link 指向其他文章、圖片有沒有 Alt Text。這些在 WriteFlow 的 Frontmatter 設計裡，有些已經被照顧到了，但實際寫作時這些規則有沒有被遵守，是我最擔心的。」

**Lisa（行銷專員）**：「Casey 說的都是技術層面，但我的視角是：這篇文章能讓讀者留下嗎？讀者點進來的第一個 10 秒鐘——標題夠不夠吸引人？開頭第一段有沒有勾子（Hook）讓他們繼續讀？有沒有清楚的 CTA 告訴讀者下一步怎麼做？我還在意文章的整體語氣——是不是跟品牌一致、是不是夠有個性？技術文章很容易寫得乾燥，但乾燥的文章不會被分享、不會被記住。我希望 AI 不只是幫我填 SEO 欄位，而是能說：『這篇文章的開頭太無聊，試試這樣改』。」

**Morgan（學習者）**：「我代表的是那些來這裡「學東西」的讀者。我打開一篇技術文章，第一個問題是：這篇文章在教我什麼？目標讀者是誰？如果文章一開始沒說清楚學習目標和前置知識，我可能讀了三分鐘才發現這個根本不適合我。我最在意的是：有沒有目錄（TOC）讓我快速掃描、有沒有程式碼範例（光說不練沒用）、有沒有總結讓我確認自己學到了什麼。還有一個常被忽略的：文章更新日期——技術文章過期了沒有？2019 年的 React 教學我不想讀。」

### 第二輪：交叉回應——三角度之間的關聯與分歧（2026-02-27）

**Casey（回應 Lisa）**：「Lisa，你說的 Hook 其實跟 SEO 有直接關係。Google 的 SERP（搜尋結果頁）Snippet 就是從 Meta Description 來的，如果 Meta Description 不吸引人，點擊率就低——而低點擊率會讓 Google 認為這篇文章對使用者不重要，進一步降低排名。所以 AI 要能分析標題和 Meta Description 的點閱吸引力，不只是確認關鍵字有塞進去就好。我希望 AI 能告訴我：『你的 Meta Description 缺乏行動訴求，試試加上「立即學習」或「5 分鐘掌握」這類詞語』。」

**Lisa（回應 Morgan）**：「Morgan 提到更新日期，我完全同意——而且這跟行銷策略有關。常青內容（Evergreen Content）是我們最想要的，但如果文章沒有定期更新標記，不只是讀者不信任，連 Google 也會降低信任度。AI 應該能分析文章裡有沒有過時的資訊特徵——比如提到某個工具的舊版本語法，或是引用了已經廢棄的 API。WriteFlow 的 Frontmatter 已有 `pubDate` 和 `lastmod`，AI 可以提醒使用者：『這篇文章已超過 6 個月未更新，建議確認內容是否仍然準確』。」

**Morgan（回應 Casey）**：「SEO 的關鍵字密度我比較不在乎，但我很在乎的是：文章有沒有把複雜概念說清楚。能不能在 5 分鐘內理解這篇文章要講什麼？AI 可以做「可讀性分析」——句子是不是太長、段落是不是太密、專有名詞有沒有解釋。這對學習者超有幫助，而且我猜這對 SEO 也有好處，因為讀者停留時間越長，Google 越喜歡。Casey 說的這一點，我們的目標其實是一致的。」

**Casey（補充）**：「Morgan 說得對，Google 的 Core Web Vitals 現在也把『使用者體驗信號』納入排名因素，讀者停留時間短、快速離開（Bounce Rate 高），都是負面信號。所以可讀性不只是學習者的需求，也是 SEO 的需求——這兩個本來以為衝突的角度，其實在很多地方是同路人。」

**Lisa（補充）**：「三個角度都在說一件事：文章要讓讀者快速判斷這篇對他有沒有用，然後讓他留下來。SEO 用的是關鍵字讓 Google 找得到，行銷用的是 Hook 讓他不關掉，學習者用的是目標聲明讓他知道值不值得花時間讀。這三件事可以整合成一個『文章開頭完整性檢查』——前 150 字有沒有同時做到：帶入關鍵字、有 Hook、說明學習目標。」

### 第三輪：聚焦——AI 應該提供哪些具體建議？（2026-02-27）

**Casey（SEO 具體建議清單）**：「把我的 SEO checklist 整理一下，AI 在分析一篇文章時應該要能檢查：

1. **標題（Title）**：是否包含核心關鍵字、長度 50-60 字元
2. **Slug**：是否包含關鍵字、全小寫無特殊字元、夠簡短（5 個單字以內）
3. **Meta Description**：長度 120-160 字元、是否有 CTA、是否有關鍵字
4. **H1 唯一性**：文章內文是否只有一個 H1
5. **H2/H3 結構**：是否有邏輯層次、關鍵字是否自然分佈在標題中
6. **內部連結**：是否有連結到其他相關文章（至少 1-2 個）
7. **圖片 Alt Text**：所有圖片是否都有填寫 Alt Text
8. **文章長度**：是否達到 700 字以上（Google 偏好深度內容）
9. **內容新鮮度**：`lastmod` 是否有更新、有無提及已過時的技術版本

建議格式要**直接、可行**——不是說『SEO 評分 60 分』，而是說：『你的 Slug 目前是 "my-first-post-2024"，建議改為 "vue3-composables-guide" 更精準也更容易被搜尋到』。」

**Lisa（行銷具體建議清單）**：「從行銷角度，我希望 AI 分析：

1. **標題吸引力**：是否包含利益點（benefit）、數字、或疑問句
2. **開頭 Hook**：前 100 字有沒有讓人想繼續讀的理由（痛點、好奇心、利益承諾）
3. **CTA 存在**：文章末尾是否有引導讀者採取下一步的行動呼籲
4. **語氣一致性**：文章語氣是否前後一致，是否符合品牌風格（專業但不失親切）
5. **分享誘因**：是否有可以讓人截圖或引用的金句、觀點、或數據
6. **視覺節奏**：文章是否太大段、缺少小標題分隔，影響讀者的視覺掃描

建議格式要用**人話**，不要說『第一段的 Engagement Potential 較低』，要說：『你的開頭直接進入技術細節，讀者還沒感覺這篇對他有用就可能關掉了——試試先說明這篇文章能幫讀者解決什麼問題』。」

**Morgan（學習者具體建議清單）**：「從學習者的角度，AI 應該檢查：

1. **學習目標聲明**：文章開頭是否說明「讀完本文，你將能夠...」
2. **目標讀者說明**：是否有說明適合什麼背景的讀者（例：「本文適合熟悉 React 基礎的讀者」）
3. **前置知識**：是否有標明需要哪些背景知識
4. **程式碼範例**：技術文章是否有附可以執行的程式碼範例
5. **目錄（TOC）**：超過 1000 字的文章是否有目錄
6. **摘要或 TL;DR**：長文是否有提供讓忙碌讀者快速掌握核心的版本
7. **術語解釋**：首次出現的專有名詞是否有解釋或附上連結
8. **資料新舊標示**：是否有明確的發布日期和最後更新日期

建議格式要直接：『這篇文章缺少學習目標聲明，建議在第一段後加上：「讀完本文，你將能夠...」，這樣讀者可以在 10 秒內判斷這篇文章是否值得繼續讀』。」

### 第四輪：整合——PM 整合三方觀點（2026-02-27）

**Alex（PM）**：「聽完三個角度，我來整理共識和行動方向。

首先，三個角色有**高度重疊**的關注點：

| 共同需求 | SEO | 行銷 | 學習者 |
|---------|-----|------|-------|
| 清楚的文章結構 | H2/H3 層次 | 視覺節奏分隔 | TOC 與層次 |
| 好的開頭 | 關鍵字在前段 | Hook 在前 100 字 | 學習目標聲明 |
| 內容新鮮度 | `lastmod` 日期 | Evergreen 標記 | 更新日期顯示 |
| Meta 資訊完整 | Meta Description | CTA 引導 | 學習目標說明 |

**不重疊但各自重要**的地方：
- SEO 獨有：Slug 精準度、圖片 Alt Text、內部連結生態
- 行銷獨有：Hook 強度、金句、語氣一致性
- 學習者獨有：程式碼範例、術語解釋、TL;DR 摘要

**與 WriteFlow 現有功能的整合**：

WriteFlow 的 Frontmatter 已有 `slug`、`description`、`keywords`、`pubDate`、`lastmod` 等欄位，Phase 1 的 SEO 生成已可輔助這些欄位。Phase 2 的文章建議功能，應以這三個角度的 checklist 為核心設計，分成三個 Section 讓使用者選擇要從哪個角度獲得建議。

**與 topic-015 決策的連結**：

本次討論的結果，直接補充了 topic-015 中 Phase 2「文章建議」Section 的功能定義。topic-015 已決定：「不做機械式評分，提供可直接使用的具體建議」——本次討論進一步確認了三個角度的具體建議格式和檢查清單。」

**Sam（Ops）**：「一個實務考量：三個角度的分析要分三次 API 呼叫，還是一次全送給 AI 讓它生成三段建議？從效能和成本角度，一次呼叫比三次好，但 Prompt 的設計需要謹慎，確保 AI 能正確分離三個角度的建議，而不是混在一起說。建議 Prompt 中明確要求 AI 以三個獨立段落分別回覆 SEO、行銷、學習者三個角度的建議。」

**Taylor（CTO）**：「Sam 說得對。架構上，建議設計成 `analyzeArticle(article, perspectives: ArticlePerspective[])` 的函式，允許指定要分析的角度。前端 UI 可以提供三個 checkbox，讓使用者選擇要分析哪些角度，然後一次呼叫 API 拿到分組結果。這樣測試容易、日後擴展也靈活——如果未來要加第四個角度（例如：無障礙性分析），只需要在 `ArticlePerspective` 型別加一個值。」

**Casey**：「還有一點值得強調：AI 的建議要有優先級。不是所有 SEO 問題都同等重要——Slug 沒有關鍵字比圖片缺 Alt Text 嚴重得多。建議 AI 輸出時用 `🔴 高優先`、`🟡 中優先`、`🟢 低優先` 標記，讓使用者知道先修哪裡。」

**Morgan**：「同意優先級的概念。從學習者角度，最重要的是前三項：學習目標、目標讀者說明、程式碼範例——沒有這三樣的文章，對我來說等於無效。其他像 TL;DR 是加分，沒有不會扣太多分。」

**Lisa**：「行銷角度，Hook 和 CTA 是最關鍵的——沒有 Hook 讀者不會繼續讀，沒有 CTA 讀者讀完就走了，什麼轉換都沒有。這兩個是 P0，其他是 P1/P2。」

## 🤝 共識與分歧

### 共識點

1. **三角度有高度重疊**：好的文章結構、清楚的開頭、內容新鮮度是所有角色都在意的
2. **AI 建議要具體、可行**：不做抽象評分，給出具體修改建議
3. **建議要有優先級標記**：`🔴 高優先`、`🟡 中優先`、`🟢 低優先`
4. **與現有 WriteFlow 功能整合**：善用 Frontmatter 欄位（slug、description、keywords、pubDate、lastmod）
5. **一次 API 呼叫**：分組建議，效能優先

### 主要分歧

- **分析深度 vs 速度**：Casey 希望全面掃描，Morgan 希望快速回饋，解決方案：分優先級
- **行銷語言 vs 直接說明**：Lisa 重視 Hook，Morgan 希望直接說清楚，解決方案：合而為一的「前 150 字檢查」

### 待釐清問題

- ❓ 一次 API 呼叫的 Prompt 設計如何確保三角度建議品質？（技術會議討論）
- ❓ 使用者可以選擇「只分析某個角度」嗎？UI 設計細節（技術會議討論）
- ❓ 對於非技術類文章（例如心情文、思想記錄），程式碼範例的建議要如何調整？
