> **出席**: Alex（PM）、Lisa（Marketing）、Jordan（User）、Sam（Ops）、Taylor（CTO）

## 背景與前情提要

2026-02-14 上次圓桌決策「品質鞏固 Sprint」以來已經 2 週。當時設定 7 個 Action Items：
- **Q-01~Q-04**：Playwright + Electron E2E 環境、三個 Happy Path 測試
- **Q-05**：Store 層 Unit Test 補齊
- **Q-06**：CI Pipeline 評估（Sam）
- **Q-07**：收集 Jordan 回饋（Alex）

原本預期：品質保護網建好後，下個 Sprint 可以安心推進功能。但實際執行有些偏差需要審查。

## 對話記錄

### 第一輪：品質 Sprint 成果回顧

**Taylor（CTO）**：我先說實話——Q-01 到 Q-05 的進度看起來不錯，但覆蓋度有缺口，特別是核心功能。Playwright 環境和 Unit Test 都建好了，143 個測試 99.3% 通過率，這部分很穩。

但三個 Happy Path 的 E2E 只完成了一半。編輯器流程（Q-02）有 2 個測試，設定路徑流程（Q-04）部分完成，但同步發布流程（Q-03）——這是整個工具的核心價值——沒看到獨立的測試，article-list 的測試全被 skip 了。

這是個問題。我們說「品質優先」，結果品質保護網最重要的一段——發布流程——居然沒有測試？

**Alex（PM）**：等等，我理解你的擔心，但我從另一個角度看。Q-01 到 Q-06 原本估 2 週，實際上 1 週左右就大致完成了。Lin 和 Sam 的效率比預期高，剩下 1 週空檔期他們沒閒著，繼續推進了一些功能開發——Sentry 整合、全文搜索、AI Service。

這不是「偏離品質 Sprint」，而是「高效利用時間」。但問題是，這導致我的 Q-07——收集 Jordan 回饋——沒跟上。我低估了測試建立的速度，高估了 Jordan 需要的時間，沒主動聯繫他。所以現在 Q-07 是缺失的。

**Jordan（User）**：嘿，你們瞎操心。我雖然沒給正式報告，但我一直在用啊。過去 2 週我發了多少文章（詳細回饋內容見 Jordan 分析報告）。

坦白講，整體體驗 7 分。最滿意的就是 Wiki Link 和圖片處理——真的省時間了。最不滿意的是自動儲存太積極，打了又刪的東西還是觸發了儲存，還有文章列表確實有點卡。

**Alex**：好，我聽到了。Jordan，根據你 2 週的實際使用，哪 3 個功能是你**最急著要**的？

**Jordan**：
1. **編輯器內直接發布** — 我不想在編輯器和文章列表頁面之間切來切去。編完了直接「發布」，完事。
2. **文章列表穩定** — 同步完之後列表會跳、排序不一致、很難找文章。這個是體驗問題。
3. **自動 Git Push** — 我知道這是下個 Sprint 的 scope，但每次同步完還要自己打 terminal，心累。

**Lisa（Marketing）**：等等，Jordan 的第一個需求——編輯器內直接發布——這個在我們的功能規劃裡都沒有呀。這不在原本的「品質 Sprint + F-06 Git 自動化」路線上。

**Taylor**：這個需求技術上可行，但不能現在做。Q-03（同步發布流程的 E2E）都沒測試呢，你還想在編輯器層加一個發布快捷方式？那等於把發布流程改了，結果測試也要改，風險倍增。

**Jordan**：不需要現在做，我就是說說哪個最想要。我現在用起來不算很不順，只是「有更順的想像」。

### 第二輪：技術債與持續改進的分歧

**Sam（Ops）**：我要插一句，Taylor 提的「發布流程沒 E2E」是對的，但我發現更關鍵的問題——**我們根本沒有生產環境監控**。

零崩潰監控、零效能監控、甚至不知道 Jordan 實際在用什麼功能。這等於 Electron 應用在完全盲飛。如果現在推功能，等於在無保護的情況下加複雜度。

**我的建議是：下個 Sprint 不只要補 Q-03 的 E2E，還要把 Sentry 監控上線。**這是 P0。不能盲目開發新功能，那樣只會埋更多炸彈。

**Taylor**：Sam 說得對。但我要澄清一下——我不是說「品質 Sprint 失敗了」，而是說「品質 Sprint 的最後一哩路還沒走完」。E2E 環境建好了，Unit Test 也做了，但發布流程這個核心功能的端到端保護，還差著。

加 Sentry 我同意，但是否 P0 還要看工時。Sentry 集成最快 4-6 小時，Q-03 補齊可能 2-3 小時。如果下個 Sprint 有 5 天，我希望至少 2 天用在補齊品質。

**Lisa**：所以你們的意思是，下個 Sprint 大概是——
- 2 天補品質（Q-03 E2E + Sentry）
- 3 天新功能（F-06 Git 自動化？還是 Jordan 說的編輯器內發布？）

**Alex**：不一定。Lisa，我有個問題。從行銷角度，F-06（Git 自動化）和 Jordan 的「編輯器內發布」，哪個更值得推廣？

**Lisa**：沒問題，我直說——**F-06 推廣價值遠高於「編輯器內發布」微調**。

F-06 完成了「一鍵發布」的整個故事。現在同步是本地動作，用戶還要自己 push——在行銷上講不清楚。但如果「同步 → 自動 commit/push → Vercel 自動部署 → 5 分鐘後文章上線」，這就是一個完整的故事，可以拍成 30 秒影片，放 Product Hunt 上。

編輯器內發布反而是「用戶體驗優化」，沒有故事性。

**Taylor**：但 F-06 風險很高。自動 Git 操作涉及用戶的資料和 repo，任何邏輯錯誤都可能摧毀 Git history。這需要備份機制、衝突檢測、操作日誌——一套完整的防護。不能靠 Unit Test 就上線。

**Sam**：Taylor 說的沒錯。F-06 需要的防護包括：
1. 自動 commit 前先備份到 `.writeflow/backups/`
2. Push 前 pull，衝突檢測立即停止
3. 所有 Git 操作記錄在審計日誌
4. 至少前 100 次操作需要用戶確認

而且 F-06 依賴 Q-03 有完整的 E2E 保護——因為發布邏輯變了，測試必須驗證新的 Git 整合不會破壞舊的同步邏輯。

### 第三輪：資源分配與優先級戰

**Alex**：聽起來下個 Sprint 有這麼幾個選項：

**選項 A**：補齊品質為主（3-4 天補 Q-03、Sentry、基本遙測），F-06 延後
**選項 B**：品質和功能平衡（2 天補品質，3 天做 F-06，但 F-06 的防護不完整）
**選項 C**：先做 F-06，品質後補（風險最高，但功能速度最快）

誰支持哪個？

**Taylor**：我只支持 A。B 和 C 都不行。F-06 沒有完整保護，等於用戶資料風險。我不就因為這個特地上次圓桌強調「還技術債」嗎？現在技術債還才還了一半，又想搶著加新功能？這是重複踩坑。

**Lisa**：我理解 Taylor 的擔心，但我也理解 Alex 的壓力——v0.1 做完這麼久了，如果 v0.2 還在修補，對外宣傳沒有新亮點。我支持 **A + 一個妥協**：下個 Sprint 把選項 A（品質優先）執行完，**同時準備 F-06 的技術設計**，讓工程師提前思考防護機制，這樣再下下個 Sprint 就能快速推進。

**Jordan**：我不在意功能速度，反正用著不急。只要別邊用邊出 bug，我就開心。我更同意 Taylor——寧願慢一點、穩一點，也不要快速做出會壞的功能。

**Sam**：Alex，從維運角度，我必須支持 A。沒監控、沒防護的功能，等於增加我維運的成本和風險。我寧願功能慢，也要求穩定和可觀測。

**Alex**：全部聽完了。我現在的狀態是——同意 Taylor 的技術判斷（品質優先），同意 Lisa 的妥協方案（設計先行），同意 Sam 的風險評估（監控是 P0）。

但我也要說一件事——團隊這兩週「隱性加速」（Sentry、全文搜索、AI Service），說明我們的執行力比預期高。問題不是執行慢，而是**沒有對齊方向**。我沒主動收 Jordan 回饋，導致功能開發和用戶需求脫節。這是我的失職。

### 關鍵共識點

1. **品質是持續速度的前提** — 測試保護網不完整，就不能安心加新功能（Taylor + Sam + Jordan）
2. **數據驅動勝於猜測** — 圓桌決策後發現「隱性功能開發」脫節，下次必須對齊用戶需求（Alex）
3. **F-06 有推廣價值，但需要防護** — 自動 Git 操作涉及用戶資料，不能跳過風險評估（Lisa + Taylor + Sam）
4. **監控系統是基礎建設** — Sentry 不是「額外功能」，而是「維運底線」（Sam + Taylor）

### 主要分歧與折衷

| 立場 | 原本主張 | 折衷方案 |
|------|---------|---------|
| **Taylor (CTO)** | 品質完全優先，暫不碰功能 | 品質優先，但同步做 F-06 技術設計 |
| **Lisa (Marketing)** | 快速推 F-06，搶市場窗口 | 同意品質優先，用設計和行銷素材準備下一波 |
| **Alex (PM)** | 平衡品質與功能 | 承認「隱性開發脫節」，下次對齐方向 |

**最終共識**：下個 Sprint 1 週內完成品質補齊 + 監控上線，之後進入 F-06 功能 Sprint，這樣 3 月中旬前可以達到「可以公開推廣的水平」。
