## 💭 Part 4: 根本原因分析

### 原因 #1: 優先級管理失敗

#### 心理因素分析

```
人性弱點: 困難任務拖延症

簡單任務（已做）:
✅ 寫單元測試 → 有成就感，立即反饋
✅ UI/UX 優化 → 視覺效果明顯
✅ Bug 修復 → 問題明確，容易解決

困難任務（未做）:
❌ 發布功能 → 涉及檔案系統、複雜邏輯
❌ Git 自動化 → 不確定性高、錯誤處理複雜
❌ 設定持久化 → 需要考慮多種場景

診斷: 逃避不確定性，選擇舒適區
```

#### 機制缺失

```
決策層面:
✅ 圓桌會議有決議 (P0 優先)
✅ MVP_SCOPE.md 有範圍定義
❌ 沒有每日檢查機制
❌ 沒有強制執行工具
❌ 開發者自行決定工作內容

結果:
- 決議形同虛設
- 優先級在執行層面失控
- 範圍蔓延無人阻止
```

### 原因 #2: 完美主義陷阱

#### 品質 vs 進度失衡

```
過度追求完美:
- 測試從 200 → 257 (+28.5%)
- 5 個 UI/UX 優化任務（非 MVP 必要）
- 友善錯誤訊息系統（MVP 可用基本版）
- ETA 顯示（錦上添花功能）

忘記 MVP 定義:
MVP = Minimum Viable Product
    = 最小可行產品
    ≠ Perfect Product

實際行為:
把 MVP 當作 Perfect Product 打造
結果: 進度嚴重落後
```

### 原因 #3: 時間估算過於樂觀

#### 估算 vs 實際對比

```
原始估算:
Wiki Link 轉換: 3 天 (已完成)
圖片路徑處理: 2 天 (已完成)
Frontmatter 轉換: 1 天 (已完成)
---------------
小計: 6 天

實際耗時: 10+ 天（包含偏離時間）

未計入因素:
❌ 非 P0 任務時間（4 天）
❌ Bug 修復時間（0.5 天）
❌ 決策討論時間（0.5 天）
❌ 文檔撰寫時間（2 天）
❌ 會議與溝通時間（1 天）

結論: 「淨開發時間」被高估 40%
```
